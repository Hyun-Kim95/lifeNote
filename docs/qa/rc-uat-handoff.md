# RC(릴리즈 후보)와 UAT(사용자 수용 테스트) 구분

## 현재 단계 판정

- 완료 아님
- QA Gate 보류 (실기기 픽셀 증빙 미완)

## 이슈 등급 요약

- **BLOCKER**: 실기기 픽셀 증빙 미완으로 RC 최종 승인 보류
- **MAJOR**: Todos 반복 규칙(`once`) null 날짜 차단 로직이 배포 전 수동 검증 필요
- **MAJOR**: 403 권한 UI 점검 결과 미기록 (권한 제한 상태 확인 필요)
- **MINOR**: 핵심 5개 화면 FIX 0 검증 체크 미완

## 다음 액션 체크리스트

- [ ] 실기기 캡처 증빙(라이트/다크 포함) 수집 후 체크리스트와 동기화
- [ ] 403 권한 UI(접근 불가/권한 제한) 동작과 카피를 화면별 점검
- [ ] 핵심 5개 화면의 FIX 잔여 0개 확인 후 Gate 재판정 요청

## 용어

| 단계 | 누가 | 의미 |
|------|------|------|
| **RC** | 개발·에이전트·내부 QA | PRD MVP 기준 제품 플로우·상태 UI·인증·API 계약 정합·증빙·알려진 제한 목록까지 완료한 빌드 |
| **UAT** | 제품 책임자(고객) | 스테이징/실기기에서 **배포 직전 최종 실사용 테스트** (기기·네트워크·실계정) |

**원칙:** 스모크용 화면(예: 수동 토큰만으로 API 호출하는 단일 화면)은 RC가 아니며, UAT 대상으로 넘기지 않는다. 수동 토큰 입력은 `__DEV__` 전용 **개발자 도구**로만 제공한다.

**PRD·Stitch·구현 정합(정적 검증):** 에이전트가 코드·문서를 대조한 결과는 [mobile-prd-stitch-verification.md](./mobile-prd-stitch-verification.md)에 정리했다. 아래 체크리스트는 **실기기·실계정 수동 확인** 완료 시에만 체크하는 것을 권장한다.

## 권장 진행 순서 (실행용)

1. 핵심 5개 화면(Home/Todos/Food/Plan/Community)부터 `mobile-stitch-pixel-checklist`로 `PASS/FIX` 기록
2. `FIX` 항목만 모아 최소 수정
3. 보조 화면(공지/상세/로그인/더보기/일기/통계/설정/커뮤니티 상세) 동일 방식 점검
4. 라이트/다크 각각 재검증 후 최종 승인

## UAT에 넘길 산출물

1. RC 빌드 설치 방법(스토어/내부 배포/EAS 링크)
2. `docs/qa/release-check-lifenote.md` 기준 수동 확인 결과(내부가 선행 수행)
3. `docs/qa/rc-handoff-{버전}.md` (또는 동등): **미구현·정책 미결·BLOCKER 아님** 항목 명시
4. 화면/핵심 플로우 증빙(캡처 또는 짧은 영상)

## 알려진 제한 (예시, 갱신 시 본문 수정)

- 단일 게시글 `GET` API 없음: 커뮤니티 상세는 목록 응답 객체를 내비게이션 파라미터로 전달한다.
- Google OAuth: `apps/mobile/.env.example`의 리다이렉트 URI를 Google Cloud 콘솔 Web 클라이언트에 등록해야 한다(API와 동일 `GOOGLE_CLIENT_ID` 사용 시).
